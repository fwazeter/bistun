# LMS-DTO: Formal Trait & Manifest Schema

> **Version:** 0.1.4

---

## I. Overview

The `CapabilityManifest` is the primary Data Transfer Object (DTO) generated by the `LinguisticManager`. It encapsulates the resolved **Typological** and **Orthographic** properties of a locale, enabling consuming services to execute generic logic for classes of languages (e.g., "Agglutinative") rather than specific language codes. This DTO is designed to be immutable once created.

---

## II. The `CapabilityManifest` DTO Schema

To prevent breaking API contracts during future updates, the DTO utilizes a dynamic Map-based structure for its core capabilities.

### 1. Root Structure

* **`resolved_locale`** (String): The BCP 47 tag resulting from the **Taxonomic** resolution and fallback process (e.g., `en-GB`).
* **`traits`** (Map<TraitKey, Object>): A keyed collection of **Typological** and **Orthographic** capabilities.
* **`metadata`** (Map<String, Object>): Observability data including registry versions and resolution performance.

---

## III. The Trait Dictionary (`TraitKey`)

Consuming services should expect the following "Golden Set" of keys in every manifest.

| Trait Key                | Value Type         | Description                                                         |
|:-------------------------|:-------------------|:--------------------------------------------------------------------|
| `PRIMARY_DIRECTION`      | Enum (`Direction`) | The UI direction derived from the primary **Orthographic** context. |
| `HAS_BIDI_ELEMENTS`      | Boolean            | True if any secondary scripts in the manifest are RTL.              |
| `REQUIRES_SHAPING`       | Boolean            | Indicates if contextual glyph shaping (e.g., Arabic) is required.   |
| `SEGMENTATION_STRATEGY`  | Enum (`SegType`)   | The boundary detection logic selected via High-Water Mark logic.    |
| `MORPHOLOGY_TYPE`        | Enum (`MorphType`) | The language's **Typological** structure (e.g., `TEMPLATIC`).       |
| `UNICODE_PRELOAD_BLOCKS` | Array<String>      | De-duplicated hex ranges for font-loading automation.               |

---

## IV. Referenced Enumerations

To ensure program-agnostic implementation, the following enum values are standardized across the service.

### 1. `Direction`

* `LTR` (Left-to-Right).
* `RTL` (Right-to-Left).
* `TTB` (Top-to-Bottom).
* `BIDI` (Bidirectional).

### 2. `SegType` (Segmentation)

* `NONE`: No automatic segmentation.
* `SPACE`: Standard whitespace-based.
* `CHARACTER`: Single grapheme cluster.
* `DICTIONARY`: Lexical analysis required (Thai, Lao).

### 3. `MorphType` (Morphology)

* `ISOLATING`, `AGGLUTINATIVE`, `FUSIONAL`, `TEMPLATIC`, `POLYSYNTHETIC`.

---

## V. Metadata and Observability

The `metadata` map provides critical context for debugging and service monitoring.

* **`registry_version`**: The Semantic Version of the registry used to generate this manifest.
* **`resolution_path`**: The sequence of tags tried during **Taxonomic** resolution (e.g., `["en-AU", "en-GB"]`).
* **`resolution_time_ms`**: Total latency of the **Taxonomic**, **Typological**, and **Orthographic** pipeline.

---

## VI. Implementation Example (JSON Representation)

```json
{
  "resolved_locale": "ar-EG",
  "traits": {
    "PRIMARY_DIRECTION": "RTL",
    "HAS_BIDI_ELEMENTS": true,
    "REQUIRES_SHAPING": true,
    "SEGMENTATION_STRATEGY": "SPACE",
    "MORPHOLOGY_TYPE": "TEMPLATIC",
    "UNICODE_PRELOAD_BLOCKS": [
      "U+0600-U+06FF"
    ]
  },
  "metadata": {
    "registry_version": "1.2.0",
    "resolution_path": [
      "ar-EG"
    ],
    "resolution_time_ms": 0.45
  }
}
```
